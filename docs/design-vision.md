# PRISM — 产品设计愿景

## 形态

PRISM 是一个横版掌机形态的随身设备，类似 GBA 横持。
MPU6050 感知姿态，设备可以横持也可以竖持，UI 自动旋转适配。

```
横持（默认）
┌─────────────────────────────┐
│  ·  [  SSD1351 128×128  ]  ·│  ← 两侧 WS2812 灯条
│     [                   ]   │
│  ↑↓←→  [OK]    [A] [B]     │
└─────────────────────────────┘
         ↑
     摇杆旁一颗 WS2812（状态灯）
```

## 灯光语言

灯光不是装饰，是信息通道。每种状态对应固定的灯光行为，用户不需要看屏幕就能感知设备状态。

| 状态 | 侧边灯条 | 摇杆旁状态灯 |
|------|---------|------------|
| 正常待机 | 极暗橙色呼吸（4s 周期） | 常亮橙色，极低亮度 |
| 音乐播放中 | 随节拍跳动，橙→白 | 随节拍闪烁 |
| 来电/通知 | 白色快闪 3 次 | 白色常亮直到确认 |
| 低电量 | 慢速红色呼吸 | 红色 |
| 蓝牙已连接 | 启动时蓝色扫过一次，之后熄灭 | 蓝色微亮 |
| 蓝牙断开 | 橙色向中间收缩一次 | 熄灭 |
| 充电中 | 从左到右流动绿色 | 绿色 |
| 充电完成 | 常亮绿色 | 绿色 |
| 深度睡眠 | 全灭 | 全灭 |
| 唤醒 | 橙色从中心向两侧展开 | 渐亮橙色 |

## 姿态交互

MPU6050 不只用于旋转屏幕，还可以作为交互维度：

- 横持 → 主界面（播放器、时钟、菜单）
- 竖持 → 快捷信息面板（通知、传感器数据、状态）
- 翻转朝下 → 静音 / 勿扰
- 快速抬起（pick-up gesture） → 点亮屏幕，显示时间
- 轻摇 → 切换下一首（需在播放器模式下确认）

姿态切换有过渡动画，不是瞬间跳变。

## 屏幕设计原则

SSD1351 128×128，AXIS UI 橙黄终端风。

**信息密度原则：主屏只显示一件最重要的事。**

- 空闲时：大字时钟 + 极简状态角标
- 播放时：专辑色块 + 曲名 + 进度条，其他全部隐藏
- 菜单时：3D 滚轮，选中项放大，其余缩小变暗

**动效原则：有重量感，不轻浮。**

- 菜单滚动：easeOutElastic，像拨动一个有阻尼的转盘
- 页面切换：水平滑动，带轻微的透视压缩
- 数字变化：从下方冒出（参考卫星时钟的实现）
- 所有动画时长控制在 120~200ms，不拖沓

## 声音缺席的补偿

PRISM 没有扬声器（音频走耳机），所有本机反馈靠屏幕 + 灯光。

规则：任何需要用户注意的事件，灯光先于屏幕响应（灯光延迟更低，更直觉）。

## WS2812 程序设计规范

灯条和摇杆旁单颗 WS2812 串联在同一条数据线上，是同一个 LED 数组。

**索引分配**

```cpp
#define LED_COUNT     N+1  // 灯条 N 颗 + 摇杆灯 1 颗
#define LED_JOYSTICK  N    // 摇杆灯在串联末尾（或头部，取决于接线）
```

焊好后先逐颗点亮确认索引顺序，再写逻辑。

**驱动分层**

```cpp
void ledSetStrip(uint32_t color);    // 控制灯条 0~N-1
void ledSetJoystick(uint32_t color); // 控制摇杆灯
void ledCommit();                    // 唯一的 show() 调用点，每帧一次
```

每帧只调用一次 `ledCommit()`，禁止在 `ledSetStrip` / `ledSetJoystick` 内部调用 `show()`。

**优先级系统**

多个状态可能同时请求灯光，高优先级持续期间低优先级请求直接丢弃：

```cpp
enum LedPriority {
    LED_PRI_IDLE    = 0,  // 待机呼吸
    LED_PRI_BEAT    = 1,  // 音乐节拍
    LED_PRI_STATUS  = 2,  // 蓝牙/充电状态
    LED_PRI_NOTIFY  = 3,  // 通知/来电（最高）
};
```

**摇杆灯职责**

它是手指的影子，亮度永远低于灯条，存在感刚好够被感知：
- 按键按下：白色闪 10ms
- 摇杆拨动：微亮一下
- 通知到达：白色急闪，覆盖其他状态
- 播放中：跟节拍微脉冲，幅度极小

## 使用场景

### 场景一：通勤半睡状态
公交车上，手攥着 PRISM，接着耳机，眼睛闭着。
拇指自然落在摇杆上，不看屏幕，靠肌肉记忆控制媒体：

- 摇杆左/右 → 上一首/下一首
- 摇杆上/下 → 音量
- A 键 → 播放/暂停
- B 键 → 保留（防误触）

切歌时摇杆旁的灯闪一下，音量变化时灯条亮度跟着变，透过眼皮可感知。
需要一个媒体锁定模式（双击 B 进入/退出），锁定后只有媒体键有效，防止手无意识用力误触菜单。

### 场景二：学习和工作时替代静音手机
桌上放着 PRISM，手机收起来。PRISM 接收来自手机的通知（BLE ANCS/GMCS），
重要通知灯条闪烁，次要通知只有摇杆灯微亮一下。
不需要拿起手机，不需要解锁，扫一眼或感知一下灯光就够了。
专注状态下可开启勿扰模式（翻转设备），所有通知静默，只记录不打扰。

### 场景三：USB 声卡
P4 通过 USB UAC（USB Audio Class）连接电脑，作为外置声卡使用。
耳机接 PRISM，PRISM 接电脑，电脑音频走 PRISM 输出。
屏幕显示当前音量、采样率等信息，摇杆控制音量，A 键静音。
这是 P4 端 USB Audio 功能的核心使用场景之一。

### 场景四：LoRa 离线短信
无 WiFi、无蜂窝网络的环境下，两台 PRISM 之间可以通过 LoRa 直接收发短消息。
地铁信号死区、户外、演出现场——只要双方在 LoRa 覆盖范围内（数百米到数公里）就能通信。
消息界面是一个独立屏幕（SCR_LORA），输入用摇杆选字或预设快捷短语。
收到消息时灯条白色扫过一次，摇杆灯常亮直到查看。

### 场景五：红外遥控家电
内置红外发射/接收模块，学习并存储家电遥控码。
在家时 PRISM 可以遥控空调、电视、灯光等。
遥控界面是一个独立的屏幕（SCR_REMOTE），按键映射可自定义。
学习遥控码时进入学习模式，对准原遥控器按键即可录入。

### 场景六：时间工具
内置 RTC 时钟模组，断电也能保持时间。主屏默认显示时钟。
独立时间工具屏幕（SCR_CLOCK）包含：

- 时钟：大字显示，AXIS UI 风格数字动画
- 秒表：A 键开始/暂停，B 键记圈，摇杆下重置
- 倒计时：摇杆上下设置时间，A 键开始，结束时灯条红色急闪
- 番茄钟：25分钟专注 + 5分钟休息，灯光区分状态（专注=橙，休息=绿）

配合勿扰模式和学习场景使用。

### 场景七：沙粒倒计时
整个屏幕是一个沙漏容器的下半部分。沙子永远从重力方向的"上方"流入，堆积在"下方"。
MPU6050 实时提供重力方向，设备怎么转，沙子就往哪里堆。

- 沙子的量 = 已流逝时间占比，时间越少沙子越多，最终几乎填满屏幕
- 数字始终浮在沙面上方，不被覆盖，像漂浮物
- 翻转设备 → 沙子全部涌向新的下方，作为重置手势

**粒子实现方案（细胞自动机）**

不需要真正的物理引擎，用简化规则模拟：
1. 每帧每颗沙粒尝试向重力方向移动一格
2. 目标格被占则尝试斜向滑落
3. 新沙粒从重力反方向边缘随机生成，速率由倒计时进度决定

ESP32-P4 跑 300~500 颗粒子无压力，单像素点表示每颗沙粒。



PRISM 是一个冷静、精准、有点神秘的设备。
它不主动打扰你，但当你需要它时，它总是准备好的。
它的 UI 不解释自己，只呈现信息。
它的灯光不炫耀，只传达状态。

> 参考气质：Teenage Engineering OP-1、早期 iPod、军用仪器面板

## 主界面设计（SCR_HOME）

### 网络节点菜单

主界面是一个网络拓扑图，每个功能是一个节点，HOME 永远在中心。

```
         [ CLOCK ]
              |
[ LORA ] ═[[ HOME ]]═ [ PLAYER ]
              |
         [ TOOLS ]
```

**节点状态**

| 状态 | 视觉 |
|------|------|
| 焦点节点 | 橙色双边框，字体稍大，内部光标闪烁 |
| 相邻节点 | 白色单边框，可到达 |
| 远端节点 | 暗灰，隐入背景 |
| 连线（焦点路径） | 亮橙 |
| 连线（其余） | 暗灰 |

**五维摇杆操作**

- 上下左右 → 光标沿连线跳到相邻节点
- OK 按下 → 进入该节点对应屏幕
- B 键（任意屏幕）→ 返回 HOME，焦点回到中心节点

**光标移动动效**

拨动摇杆时，一个小点从当前节点沿连线滑向目标节点，80ms 到达，焦点切换。快而有重量感。

### 消息气泡

手机通知（BLE ANCS/GMCS）或 LoRa 消息到达时，气泡从底部冒出：

```
┌──────────────────────────┐
│                          │
│        [ CLOCK ]         │
│             |            │
│  [ LORA ]═[[HOME]]═[ PLAYER ]
│             |            │
│        [ TOOLS ]         │
│                          │
│  ╭────────────────────╮  │
│  │ ● WECHAT           │  │
│  │   你好啊           │  │
│  ╰────────────────────╯  │
└──────────────────────────┘
```

- 气泡从底部 easeOutElastic 弹出，节点图整体上移几像素让位
- 左侧圆点表示来源：BLE 通知=白，LoRa=橙
- 停留 4 秒后向下滑出，节点图回到原位
- 多条消息排队，不叠加
- 气泡用圆角 `╭╮╰╯`，与节点图直角 `[]` 形成软硬对比，有层次感

## 后台服务架构

媒体和通知是服务，不是屏幕。屏幕只是它们的控制面板和展示窗口。

**32E 端（永远在线，后台服务层）**
- A2DP sink 持续接收蓝牙音频，不受 P4 屏幕切换影响
- BLE ANCS/GMCS 持续监听手机通知，收到后通过 UART 推给 P4
- 传感器数据（MPU6050、温湿度、气压）持续采集，P4 按需拉取
- 32E 不关心当前显示什么，只管服务持续运行

**P4 端（按需唤醒，前台 UI 层）**
- 屏幕切换只影响显示层，不影响任何后台服务
- SCR_PLAYER 是播放器的控制界面，不是音频宿主
- 通知气泡是独立的覆盖层，任何屏幕都能收到
- USB Audio（UAC）作为独立服务运行，连接电脑时自动激活

**UART 消息协议（32E → P4）**
32E 主动推送事件，P4 被动接收并更新 UI：
- 通知到达 → 推送气泡数据
- 媒体状态变化（曲目、进度、播放状态）→ 推送更新
- 传感器数据 → 定时推送
- 蓝牙连接状态变化 → 推送事件

**后台定时器**

定时器、番茄钟、秒表作为后台服务运行，退出 SCR_CLOCK 后继续计时：
- CLOCK 节点旁显示小橙点，表示有定时器在运行
- 时间到：灯条红色急闪 + 气泡弹出"TIMER DONE"，按任意键确认
- 重新进入 SCR_CLOCK 可查看状态或重置
